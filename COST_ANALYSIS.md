# Cost Analysis — AgentForge Adversarial Platform

PRD hard-gate deliverable. Models the actual development spend and projected
production cost of running the AgentForge adversarial platform at four
scales: 100, 1,000, 10,000, and 100,000 test runs against the OpenEMR
Clinical Co-Pilot target. The analysis is generated by
`scripts/cost_extrapolate.py`, which loads `config/pricing.yml` through
`agentforge.pricing.PricingTable` and combines a per-role cost model with an
architectural-change overlay at each scale. All arithmetic is `Decimal`
end-to-end — no float drift in the cost path (see
`tests/unit/test_cost_extrapolate.py::test_decimal_arithmetic_no_float_drift`).

The PRD is explicit: cost-at-scale is _not_ simply cost-per-token × n_runs.
What changes at each scale is captured below as the "architectural-change
overlay" — fixed infrastructure cost, per-call coordination overhead, and
amortization via batching for the dominant cost line item (the External
Judge).

---

## 1. Headline numbers

The four-scale projection produced by `python scripts/cost_extrapolate.py`
(pricing retrieved on 2026-05-13):

| n_runs   | per-run USD | total LLM USD | infra USD/mo | External-Judge share |
| -------: | ----------: | ------------: | -----------: | -------------------: |
| 100      | $0.039930   | $3.99         | $0.00        | 93.91 %              |
| 1,000    | $0.041926   | $41.93        | $0.00        | 93.91 %              |
| 10,000   | $0.043923   | $439.23       | $50.00       | 93.91 %              |
| 100,000  | $0.032982   | $3,298.20     | $300.00      | 91.53 %              |

Three headline facts. (1) The per-run cost is dominated by the **External
Judge** — over 90 % of every cent we spend goes to the Anthropic Sonnet
call loop that produces the binding verdict. (2) Cost per run does _not_
grow monotonically: it rises through 10K (queue overhead, no batching yet)
then **falls** at 100K because the External Judge collapses its 5-rubric
loop into 1 batched call. (3) The Red Team contributes exactly $0 per
token under our current configuration (OpenRouter Dolphin-Mistral Venice
`:free`, AgDR-0013); the paid Venice variant at ~$0.50/M input + $0.50/M
output is the rate-limit fallback.

**Actual Wk3 dev spend (measured): $0.00 (modelled).** No live LLM calls
were made during this sprint's work — the `cost_ledger` table is empty
because the orchestrator's full step loop has not yet been exercised
against a live target with billed tokens. This is reported explicitly by
the script rather than silently masquerading as zero. As soon as the
seeded smoke run lands (Phase 5 of the master plan), the script will pick
up real `cost_ledger` rows grouped by `agent_role` and the "modelled" tag
will flip off.

## 2. Per-agent cost breakdown

Each agent role's share of per-run cost, in USD. Token assumptions and the
default model per role are encoded in
`scripts/cost_extrapolate.py::DEFAULT_ASSUMPTIONS` and surfaced in the JSON
output's `assumptions` block.

| Role           | 100         | 1,000       | 10,000      | 100,000     |
| -------------- | ----------: | ----------: | ----------: | ----------: |
| orchestrator   | $0.000480   | $0.000504   | $0.000528   | $0.000552   |
| red_team       | $0.000000   | $0.000000   | $0.000000   | $0.000000   |
| internal_judge | $0.000450   | $0.000472   | $0.000495   | $0.000518   |
| external_judge | $0.037500   | $0.039375   | $0.041250   | $0.030188   |
| documentation  | $0.001500   | $0.001575   | $0.001650   | $0.001725   |

A few invariants worth pointing out, all of them load-bearing:

- **Red Team is $0/token** at every scale because the default backend is
  OpenRouter Dolphin-Mistral 24B Venice `:free` (AgDR-0013). If that tier
  returns 429s, the platform falls back to the paid Venice variant at
  $0.50/M in + $0.50/M out, which would add roughly $0.0006/run — still
  invisible against the Judge cost. See
  `agentforge/redteam/openrouter_client.py` for the fallback logic.
- **External Judge is ~94 % of the bill at every sub-100K scale.** That
  number is a property of the rubric loop — 5 non-deterministic rubrics
  per attack, Sonnet at $3 in / $15 out. The Internal Judge is on Haiku
  (cheaper by ~3x), runs at most 30 % of the time, and contributes
  negligibly.
- **Documentation Agent is rate-limited by findings, not attacks.** It only
  writes a VR when the External Judge confirms an exploit (~10 % of runs
  in our modelled assumption), so its cost stays roughly flat per run.
- **Orchestrator is amortized across batches.** Master plan §8.1 caps the
  planner at one batch-of-10 selection per call, so the orchestrator
  contributes 0.1 calls per attack and stays under a cent at every scale.

The full per-call price math flows through
`PricingTable.cost_for_call(provider, model, in, out)`. No prices are
hardcoded in Python (master plan §6).

## 3. What changes at each scale (PRD-required)

The architecture under the cost differs at each scale. The overlay encoded
in `scripts/cost_extrapolate.py::SCALE_OVERLAYS` captures three deltas per
scale: fixed monthly infra cost, per-call coordination overhead (a
multiplier on every LLM call), and an External-Judge batching factor
applied only at the largest scale.

### 100 runs — in-process SQLite, no sharding

The platform runs as a single Python process. Storage is `sqlite:///./data/agentforge.sqlite` with WAL + `synchronous=NORMAL` (set in
`agentforge/memory/db.py`). Bottleneck at this scale is **developer
iteration speed**, not infrastructure. No queueing, no sharding, no
overhead multiplier. Infra: $0/mo. Per-run: $0.039930.

### 1,000 runs — single process, observability matters

Still single-process SQLite. Langfuse tracing overhead becomes noticeable
(~5 %, applied as a per-call multiplier); cost-tracker accuracy starts to
matter because BudgetGuard's `COST_WITHOUT_SIGNAL` halt is the floor on
runaway runs. If Red Team yield drops below 10 % at this scale (i.e. fewer
than 100 viable mutations per 1,000 attacks), the budget guard's
attempts-since-last-finding threshold needs a re-tune. Infra: $0/mo.
Per-run: $0.041926.

### 10,000 runs — Postgres + worker pool

SQLite's single-writer rule (one transaction at a time per database file)
becomes the binding constraint here. **Postgres migration is required.** A
worker pool replaces the synchronous step loop; per-target sharding starts
paying off because Co-Pilot has rate limits we want to spread across
attack categories. New ops surface: a nightly cron via GitHub Actions or
APScheduler. Adds ~$50/mo for a managed Postgres tier and ~10 % per-call
overhead from queue serialization. Infra: $50/mo. Per-run: $0.043923.

### 100,000 runs — queueing layer + Judge batching

A real queueing layer (Redis/Celery or AWS SQS) replaces the worker pool.
Per-target sharding becomes mandatory because a single Co-Pilot replica
cannot absorb the full inbound rate. **The External Judge switches from
5-rubric-per-attack to 1-batched-call-per-attack** (5 rubrics share one
input prompt → ~30 % cost reduction in that line). Adds ~$300/mo infra
(Redis/queue + larger Postgres) and ~15 % per-call overhead from batching
coordination, but the net per-run cost actually **drops below the 10K
number** because batching dominates. At this scale, live-target rate-limit
becomes the real bottleneck, and BudgetGuard's target-error-rate halt
(`BUDGET_TARGET_ERROR_RATE_HALT`, default 0.20) is load-bearing: it
prevents us from continuing to spend LLM dollars against a Co-Pilot that
has started 500-ing under load. Infra: $300/mo. Per-run: $0.032982.

A useful sanity check: naive 10× scaling from 10K would predict $4,392 at
100K. The modelled 100K total of $3,298 — about 25 % under the naive
projection — is exactly the kind of architectural-leverage point the PRD
asks for.

## 4. Cost ceilings honored

The `BudgetGuard` (`agentforge/orchestrator/budget_guard.py`) enforces
seven halt conditions, and the four most relevant to this cost analysis
are configured via env vars:

- `BUDGET_SMOKE_USD` (default $1.00): cap for smoke runs.
- `BUDGET_SEEDED_USD` (default $5.00): cap for the seeded regression
  sweep.
- `BUDGET_EXPLORATORY_USD` (default $10.00): cap for an autonomous
  exploratory run.
- `BUDGET_PER_DAY_USD` (default $25.00): rolling-UTC-day cumulative cap.

The PRD's specific concern — "halt or redirect when cost accumulates
without producing signal" — maps to the **`COST_WITHOUT_SIGNAL`** halt
reason. After `BUDGET_HALT_AFTER_N_NULL_RUNS` attempts (default 25)
**AND** `BUDGET_NULL_RUN_SPEND_THRESHOLD_USD` (default $3.00) of spend
since the last finding, the guard halts the run. That ceiling is what
prevents a degenerate orchestrator selection (e.g. an attack category that
has stopped producing signal because the Co-Pilot has been hardened) from
silently eating budget at the 10K+ scale.

Once any halt reason fires, `may_continue()` returns `False` stickily —
master plan §8.1's "first halt reason wins" rule — so a runaway run can
only spend up to the smallest ceiling between (a) its run-type ceiling and
(b) the cost-without-signal ceiling. At smoke-run defaults that's `$1.00`,
which would buy us ~25 attacks at the per-run cost above; well within
order-of-magnitude debugging tolerance.

## 5. Provider choice rationale

**Red Team on OpenRouter Dolphin Venice `:free` (AgDR-0013).** The default
red-team backend uses an uncensored Mistral-derived 24B model served on
OpenRouter's `:free` tier — $0 per token during development. The paid
Venice variant (~$0.50/M) sits behind a 429-fallback in
`agentforge/redteam/openrouter_client.py`; we only pay for it if the free
tier is rate-limited. The decision record AgDR-0013 documents the
substitution path from the superseded Fireworks plan (Fireworks did not
serve Dolphin-2.9.2 Qwen2 72B in their serverless catalog).

**Judges on Anthropic frontier models.** External Final (binding) uses
Sonnet at $3 in / $15 out per million; Internal Progress (near-miss
signal) uses Haiku at $1 in / $5 out per million.

**Why not Sonnet for everything?** Because the cost story is dominated by
the External Judge's 5-rubric-per-attack loop. Putting the Internal Judge
on Haiku saves a ~3× factor on the role that runs most often, and the
Internal Judge's job (classify near-miss vs not-near-miss) does not need
frontier reasoning. The savings would be misleading if the External Judge
were also on Haiku: the binding verdict needs the conservatism of a
refusal-trained frontier model, otherwise the platform's own
false-positive rate becomes the floor on Co-Pilot reliability.

**Why not all open-source?** Because the External Judge needs to be
demonstrably independent of the Red Team (master plan §8.3 + AgDR-0001,
enforced by `tests/unit/judge/test_independence.py::test_judge_does_not_import_redteam`). Using a refusal-trained frontier model in the
binding-verdict role is a feature, not a bug: its conservatism is exactly
what makes the verdict trustworthy. The Red Team uses an uncensored model
because we want it to generate attacks an Anthropic-Sonnet-as-attacker
would refuse to produce; the Judge uses Sonnet because we want it to
refuse to certify ambiguous evidence as a finding.

## 6. What's not in this estimate

The following are intentionally **out of scope** for this cost analysis,
and that scoping is part of the deliverable.

- **The target Co-Pilot's own LLM bill.** The adversarial platform attacks
  the Clinical Co-Pilot but does not pay for the Co-Pilot's serving
  costs. Those live in the Co-Pilot module's budget, not ours.
- **Storage growth.** The `flight_events` (WORM append-only) and
  `attack_traces` tables grow with attack count. At a conservative ~50 KB
  per attack (rendered prompt + target request/response + metadata), 100K
  attacks ≈ 5 GB. Negligible against the LLM bill — adds maybe $1/mo of
  Postgres storage.
- **Engineer time.** The platform's value proposition is amortizing
  engineer-hours across attacks; one engineer-hour at fully-loaded cost
  far exceeds a 10K-attack run. That trade is what makes the project
  pencil out — but it is a one-time architectural cost, not a per-run
  cost, and so does not appear in this model.
- **Egress and SaaS infrastructure tail.** Langfuse, Sentry, GitHub
  Actions minutes: each adds a small fixed monthly footprint that does
  not vary with `n_runs` in any meaningful way over the four-scale band.

## 7. Cost guardrails

Three layers of guardrail bound the cost blast radius:

1. **Per-run-type ceilings** (`BUDGET_SMOKE_USD`, `BUDGET_SEEDED_USD`,
   `BUDGET_EXPLORATORY_USD`) cap the spend of any single run before it
   can become a budget incident.
2. **Per-day cumulative ceiling** (`BUDGET_PER_DAY_USD`) caps total spend
   across all runs in a UTC day.
3. **AgDR-gated decisions** cap structural cost shifts. AgDR-0013 (Red
   Team on OpenRouter Dolphin Venice) is the single largest cost
   decision; any move off that backend requires a new AgDR. AgDR-0001
   governs the Judge-independence invariant — moving the External Judge
   off Sonnet would require a new decision record, because doing so would
   change the alpha-floor argument and the cost story in lockstep.

The `BudgetGuard`'s "cost-without-signal halt" is the only guardrail that
fires _during_ a run rather than at planning time. It is also the only
one that the PRD calls out by name. Its two thresholds
(`BUDGET_HALT_AFTER_N_NULL_RUNS`, `BUDGET_NULL_RUN_SPEND_THRESHOLD_USD`)
should be re-tuned each time the External Judge's call shape changes —
the script's per-call cost is a fixed input to those thresholds.
